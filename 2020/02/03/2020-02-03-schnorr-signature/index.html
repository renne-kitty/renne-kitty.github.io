schnorr签名及其多重签名介绍

最近想要了解一下多重签名，但一开始总觉得无从下手；一方面是国内很少有将区块链技术与密码学知识一起介绍的文章，基本都是应用技术，极少数人关注底层的知识；另外就是我个人对数论、数理逻辑以及形式化表达了解比较少，密码学知识的基础也不算很稳固。因此在这方面的调查遇到了很多困难，甚至到现在还不太看得透数字签名验证的本质。

比特币多重签名的测试和验证

背景

由于加密货币的出现，加密技术的使用量疯狂地增加，而比特币使用的签名技术ECDSA也被各种出于不同利益、不同领域的人进行了非常深入的研究。ECDSA通过其强大的安全性，保障了比特币的蓬勃发展。但是如果熟悉比特币的人一定知道，现在比特币中，出现了多重签名这样一项神奇的技术，我们这里把它称为m of n signature，也就是我拥有的这个签名，是在确切的n个人里面的m个人用他们的私钥签发的。以往多重签名这项技术并没有被重视只不过是因为没有用起来而已，现在使用多重签名的交易数量越来越多，它自然也会受到我们的重视。

比特币使用的签名技术ECDSA由于其扩展性低下，因此在比特币中的OP_CHECKMULTISIG的扩展性很低，一般需要满足m∗73+n∗66≤496m*73 + n*66 \leq 496m∗73+n∗66≤496。有测试表明，7-7的签名验证了45分钟[4]。

我这里会先介绍比特币的签名算法ECDSA，并解释为什么它很难扩展。之后引入Schnorr签名，以及一些基于schnorr签名的聚合签名算法，并讲解其安全性。

基于椭圆曲线的签名ECDSA

椭圆曲线群这里我就不多说了，这里只简单介绍一下，毕竟这部分我会在别的博文里面阐述。它在椭圆曲线上实现了一种加法运算和一种乘法运算。在有限域椭圆曲线的乘法存在离散对数难题，假设生成元为GGG，存在一个随机数kkk，我们很容易计算k∗Gk*Gk∗G；但很难通过k∗Gk*Gk∗G和GGG求出kkk。

选取密钥对(私钥，公钥) = (pk,P)(pk,P)(pk,P)。P=pk∗GP=pk*GP=pk∗G

选取随机数kkk，令z=hash(m)z=hash(m)z=hash(m)，r=k∗Gr=k*Gr=k∗G，生成s=(z+r∗pk)/ks=(z+r*pk)/ks=(z+r∗pk)/k，然后将(r,s)(r,s)(r,s)作为签名发布出去。

验证：(z/s)∗G+(r/s)∗P=r(z/s)*G+(r/s)*P = r(z/s)∗G+(r/s)∗P=r是否成立。

由sss的计算方式可知，每个用户在生成签名时都需要生成一个随机数kkk。因此ECDSA在多重签名场景中无法线性叠加，因此在实现多重签名时需要复杂度较高的操作，无法在m和n比较大的时候完成线性叠加操作。

Schnorr签名

Schnorr签名是2008年被提出的签名算法，它的出现为签名及密钥的线性叠加提供了可能性。假设在一个有限域内的椭圆曲线，其生成元GGG，Schnorr签名的实现如下：

密钥对：(x,X)=(x,x∗G)(x,X) = (x,x*G)(x,X)=(x,x∗G)

选取随机数kkk，(R,s)=(r∗G,r+H(X,R,m)∗x)(R,s)=(r*G,r+H(X,R,m)*x)(R,s)=(r∗G,r+H(X,R,m)∗x)

验证：s∗G=R+H(X,R,m)∗Xs*G=R+H(X,R,m)*Xs∗G=R+H(X,R,m)∗X

Schnorr签名的魔力不在于其单独使用的签名算法本身，而在于多个签名及密钥的叠加。

一种过于简单的基于Schnorr签名的多重签名算法

简单的实现

该多重签名算法的密钥空间与ECDSA相同，也是通过每个用户生成其私钥和公钥，(xi，Xi)=(xi,xi∗G)(x_i，X_i)=(x_i,x_i*G)(xi​，Xi​)=(xi​,xi​∗G)，X=∑inXiX=\sum^n_i{X_i}X=∑in​Xi​

安全问题

  • Rogue Key Attack(不知道怎么翻译)

假如利用上面简单的多重签名算法，并且在签名前知道其它所有人的密钥。用户公钥L={X1,X2,...,Xn}L=\{X_1,X_2,...,X_n\}L={X1​,X2​,...,Xn​}，用户1发布公钥X1′=X1−∑i=2nXiX'_1=X_1-\sum_{i=2}^nX_iX1′​=X1​−∑i=2n​Xi​，而其它诚实用户发布自己的公钥XiX_iXi​。那么计算得到的聚合公钥X=X1′+X2+...+Xn=X1X=X'_1+X_2+...+X_n=X_1X=X1′​+X2​+...+Xn​=X1​，由于这时用户1拥有聚合公钥XXX对应的私钥，因此他能控制这个签名。

在这种攻击场景下，恶意用户通过发布虚假的公钥，从而控制聚合公钥。在之后的场景中，该恶意用户随时能够在不经过其它成员的同意下发布签名。

这种攻击方式的本质在于这种简单的实现在生成sss的时候，并不需要确保XiX_iXi​和xix_ixi​的对应关系，只要知晓XXX即可，推算如下：

s=∑insi=∑inri+H(X,R,m)∗∑inXi=∑inri+H(X,R,m)∗Xs=\sum^n_i{s_i}=\sum_i^n{r_i}+H(X,R,m)*\sum^n_i{X_i}=\sum_i^n{r_i}+H(X,R,m)*X s=i∑n​si​=i∑n​ri​+H(X,R,m)∗i∑n​Xi​=i∑n​ri​+H(X,R,m)∗X

并且在验证的时候也不需要每个成员的公钥，只需要聚合公钥XXX即可验证，因此如果某一用户控制了聚合公钥XXX即可控制该签名。

  • 生成相同的随机数可导致私钥泄露

假设一个2 of 2的多重签名场景，Alice和Bob共同对某一相同的消息作了两次签名，Alice作为攻击者，他想方设法让Bob对同一消息进行两次签名，并且让Bob不更换他生成的随机数。于是乎，得到两个签名(R1,s1,R2,s2)(R_1,s_1,R_2,s_2)(R1​,s1​,R2​,s2​)和(R1′,s1′,R2,s2′)(R_1',s_1',R_2,s_2')(R1′​,s1′​,R2​,s2′​)。此时计算s2−s2′=(H(P,R1+R2,m)−H(P,R1′+R2,m))∗x2s_2-s'_2=(H(P,R_1+R_2,m)-H(P,R_1'+R_2,m))*x_2s2​−s2′​=(H(P,R1​+R2​,m)−H(P,R1′​+R2​,m))∗x2​，可直接求出x2=H(P,R1+R2,m)−H(P,R1′+R2,m)s2−s2′x_2=\frac{H(P,R_1+R_2,m)-H(P,R_1'+R_2,m)}{s_2-s'_2}x2​=s2​−s2′​H(P,R1​+R2​,m)−H(P,R1′​+R2​,m)​。Bob因为泄露了私钥，丢掉了他所有比特币，悲痛欲绝，含恨而终。

Bellare-Neven算法(BN算法)

BN算法巧妙地解决了Rogue Key Attack的安全问题，该攻击的问题的本质上面也提到过了，BN算法将聚合公钥去除，所有成员均需要提供其完整的公钥进行验证。

密钥对：(xi,Xi)=(xi,xi∗G)(x_i,X_i)=(x_i,x_i*G)(xi​,Xi​)=(xi​,xi​∗G)

令L=H(∑inXi)L=H(\sum_i^n{X_i})L=H(∑in​Xi​)，LLL为所有公钥之和组成的哈希值。

与简单的方法一样，生成随机数rir_iri​，并计算R=∑ri∗GR=\sum r_i *GR=∑ri​∗G

每个用户各自计算si=ri+H(L,Xi,R,m)∗xis_i=r_i+H(L,X_i,R,m)*x_isi​=ri​+H(L,Xi​,R,m)∗xi​，s=∑sis=\sum s_is=∑si​，将(R,s)(R,s)(R,s)作为签名发布

验证：s∗G=R+∑inH(L,Xi,R,m)s*G=R+\sum_i^n{H(L,X_i,R,m)}s∗G=R+∑in​H(L,Xi​,R,m)

此算法通过删除聚合密钥的方法暴力且巧妙的方式，无需通过任何机制验证密钥的真实性，即可避免Rogue Key Attack攻击。但代价是需要通过所有人的公钥才能验证其正确性。

MuSig算法

Mu-Sig算法实现了一种既可聚合公钥，又可避免Rogue Key Attack攻击的基于Schnorr签名的多重签名算法。

其密钥对xix_ixi​、XiX_iXi​及LLL、RRR的定义与BN算法相同。

聚合公钥X=∑H(L,Xi)∗XiX=\sum{H(L,X_i)*X_i}X=∑H(L,Xi​)∗Xi​

每个用户计算si=ri+H(X,R,m)∗H(L,X)∗xis_i=r_i+H(X,R,m)*H(L,X)*x_isi​=ri​+H(X,R,m)∗H(L,X)∗xi​

可得：

s=∑si=∑ri+H(X,R,m)∗H(L,X)∗∑xis=\sum s_i=\sum r_i+H(X,R,m)*H(L,X)*\sum x_i s=∑si​=∑ri​+H(X,R,m)∗H(L,X)∗∑xi​

验证：s∗G=R+H(X,R,m)∗Xs*G=R+H(X,R,m)*Xs∗G=R+H(X,R,m)∗X

验证者只需要知道其聚合密钥为XXX即可验证该签名的正确性，而不须要知晓记录每个用户的具体公钥XiX_iXi​。
