arch linux 扩展显示器折腾记

症状描述

在一个月前配置好arch以后，从来没有把电脑拿出去用过，今天把HDMI接口拔掉之后，发现无法显示图形界面，而把HDMI接口接上之后则可以正常显示在扩展显示器上。当HDMI接口拔掉之后，开机界面显示了Reached target Graphical Interface，而运行xrandr则提示Can't open display

折腾

一旦发生这种问题，第一反应就是驱动问题，所以重新安装驱动。

安装好之后完全没卵用，于是考虑使用其他的驱动，试了开源驱动nouveau、intel卡核显驱动后也是完全没卵用。而且xrandr仍然是Can't open display，这里就陷入了沉思，怀疑是X server没有识别到n卡驱动，然后继续google慢慢找了。

正确的道路

再次确认有没有识别到自己集显和独显，输出的结果说明找到了，而且独显的bus info是pci@0000:01:00.0，这个等一下要输入到xorg配置文件里

xorg配置文件生成

看到一份别人的xorg配置文件，看到别人比我的多了BusID一项。

加了之后还是没什么用，于是继续google

将n卡的开源驱动加入黑名单

由于n卡的闭源驱动和开源驱动是冲突的，想要实现扩展显示器，必须将开源驱动禁掉。

重新安装集显驱动

重新看一遍arch wiki中NVIDIA Optimus条目

由于笔记本的显示屏默认连接intel核显的，而外接显示器一般是连接独显的，所以没有安装intel卡的驱动当然不行了（为什么现在才想到。。。）

将设置加入SDDM的设置中

更新内核

安装集显之后，还是不行，然后考虑别的因素，查看一下日志（其实一直都有看，只不过对之前的阶段没什么判断性的意义罢了）

/var/log/Xorg.0.log其中有一行，“NVIDIA: Failed to initialize the NVIDIA kernel module”。总算找到突破口了，于是就找为什么会fail to init，既然是和内核相关的东西就用dmesg查看。

得到结果：“nvidia: version magic ‘5.2.0…’ should be ‘5.1.7… SMP mod_upload …’”，看到这两个数字后，其他的什么都不用看了，5.1.7是之前安装时的内核版本，然后5.2.0是现在的内核版本，滚动一下顺带更新一下内核就解决了。

重新安装中文字体

安装好内核后，重启电脑，扩展显示器已经正常启动。但看到KDE桌面、Konsole、Dolphin均有部分乱码的现状。

首先考虑环境换量问题，终端输入locale命令，结果如下：

查看locale.gen中zh_CN.UTF-8和en_US.UTF-8均已取消注释，故可排除环境变量问题。把思路逆转过来，不是所有中文都是乱码的，还有部分没乱码的中文存在，那部分的中文是楷体的，跟以前的不同，故继续考虑字体的问题。

重新安装中文字体wqy-zenhei

KDE乱码的情况得到解决，问题到这里暂时全部解决。从下午1点，一直搞到1点，总算搞完了，搞个驱动都弄得很有成就感似的，上面的东西很多是凭记忆和浏览器历史写出来的，难免会和当时情况有出入。
